The API returns consistent error envelopes

An error helper is like a standard incident report form. Instead of every route inventing a different failure shape, callers can expect message, code, and status.

CODE 
if (!sortResult.success) {
  return jsonError("Invalid sort parameter", "VALIDATION_ERROR", 400);
}

if (
  category &&
  !ENTRY_CATEGORIES.includes(category as (typeof ENTRY_CATEGORIES)[number])
) {
  return jsonError("Invalid category", "VALIDATION_ERROR", 400);
}
PLAIN ENGLISH

If the sort option is not one of the accepted values, stop.

Return a clear validation error instead of guessing.

If a category filter exists, check it against the known category list.

Reject unknown categories before asking the database for records.

The best debugging prompts are specific

When AI gets stuck, give it the layer and symptom. That keeps it from randomly changing UI when the real bug is in the route, validation, or database query.

401

Unauthorized

Inspect middleware, Clerk auth, and protected API routes.

400

Validation error

Inspect Zod schemas, date parsing, metric type, and category lists.

DB

Missing records

Inspect userId scoping, metric filter, sort order, and Prisma migrations.

DBG
Practical rule

Do not ask AI to "fix the dashboard" first. Say what broke, where you saw it, and which layer you suspect.
