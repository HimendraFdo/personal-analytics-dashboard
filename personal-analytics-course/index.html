02

Meet the Actors

A good codebase is easier to steer when you know who is responsible for what. This app has a small cast with clear jobs.

The file tree is the map

Think of the project like an office building. Each floor has a purpose: pages greet visitors, components do visible work, hooks coordinate state, and lib files handle shared rules.

app/Next.js pages, layouts, loading states, and API routes
(shell)/Protected dashboard, entries, analytics pages
api/entries/Server endpoints for entry data
components/Visible dashboard pieces like forms, lists, cards, and charts
contexts/Shared entry state for every page inside the shell
hooks/Browser logic for loading entries and choosing a metric
lib/API helpers, validation, analytics, Prisma, nutrition math
prisma/Database shape and migrations

The actors talk like a group chat

When something changes, these pieces do not all mutate the database directly. They pass messages through the right channels.

P
Page

I need entries for the current metric before I can render.

C
EntriesContext

I provide loading, error, entries, and mutation functions to every protected page.

H
useEntries

I call fetchEntries, createEntry, updateEntry, and deleteEntry, then update browser state.

A
API Route

I check auth, validate input, and talk to Prisma.

D
Database

I store only the entries that belong to the signed-in user.

The provider is the meeting room

A context is useful when many pages need the same live data. Here, it prevents dashboard, entries, and analytics from each fetching separate copies in separate styles.

CODE 
export function EntriesProvider({ children }: { children: ReactNode }) {
  const value = useEntries();
  return (
    <EntriesContext.Provider value={value}>{children}</EntriesContext.Provider>
  );
}

Metrics Change the Whole App

The metric tab is a lens. Switch from time to money to calories and the same entry system changes labels, colors, default categories, formatting, and analytics.

One route, three personalities

The app stores the active metric in the URL query string. That means a link like ?metric=money can reopen the dashboard in the money view.

CODE 
const activeMetric = parseMetricType(searchParams.get("metric"));

const setActiveMetric = useCallback(
  (metricType: MetricType) => {
    const nextParams = new URLSearchParams(searchParams.toString());
    nextParams.set("metric", metricType);
    router.push(`${pathname}?${nextParams.toString()}`);
  },
  [pathname, router, searchParams]
);
PLAIN ENGLISH

Read the current metric from the page URL.

When the user chooses a tab, copy the existing query parameters.

Replace only the metric value.

Navigate to the same page with the new metric in the URL.

The API returns consistent error envelopes

An error helper is like a standard incident report form. Instead of every route inventing a different failure shape, callers can expect message, code, and status.

CODE 
if (!sortResult.success) {
  return jsonError("Invalid sort parameter", "VALIDATION_ERROR", 400);
}

if (
  category &&
  !ENTRY_CATEGORIES.includes(category as (typeof ENTRY_CATEGORIES)[number])
) {
  return jsonError("Invalid category", "VALIDATION_ERROR", 400);
}
PLAIN ENGLISH

If the sort option is not one of the accepted values, stop.

Return a clear validation error instead of guessing.

If a category filter exists, check it against the known category list.

Reject unknown categories before asking the database for records.

The best debugging prompts are specific

When AI gets stuck, give it the layer and symptom. That keeps it from randomly changing UI when the real bug is in the route, validation, or database query.

401

Unauthorized

Inspect middleware, Clerk auth, and protected API routes.

400

Validation error

Inspect Zod schemas, date parsing, metric type, and category lists.

DB

Missing records

Inspect userId scoping, metric filter, sort order, and Prisma migrations.

DBG
Practical rule

Do not ask AI to "fix the dashboard" first. Say what broke, where you saw it, and which layer you suspect.
